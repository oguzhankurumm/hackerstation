HackerStation AI System
Local 8GB Optimized Architecture

Self-healing local AI router with memory-aware runtime, auto-repair loop, and graceful degradation under pressure.

100% local · zero cloud Apple M3 · 8GB · Metal 4 stdlib-only Python stability > performance

1 · Overview

HackerStation is an offline AI workstation tuned for an 8GB Apple Silicon box. Two specialised local models sit behind a thin Python router that classifies each request and forwards it to the right model — with a supervisor watching the whole stack, ready to restart, fall back, or throttle when pressure builds.

Models

2 · How it works

Routing logic

The router uses two compiled regex sets to score the prompt for coding-flavoured vs reasoning-flavoured keywords. Higher score wins; on tie, the lighter coding model is chosen.

coding_score    = len(CODING_PATTERNS.findall(prompt))
reasoning_score = len(REASONING_PATTERNS.findall(prompt))

if coding_score > reasoning_score:    return "coding"
if reasoning_score > coding_score:    return "reasoning"
return "coding"   # tie → cheaper model

Force-routing is available via dedicated endpoints:

  • POST /generate/coding — always coding model
  • POST /generate/reasoning — always reasoning model
  • POST /generate — auto-route

Memory-constraint design

Every choice in the router exists because of the 8GB limit:

  • One in-flight generation — a Semaphore(1) keeps two heavy inferences from colliding and OOM-killing the box.
  • Reduced context window — num_ctx=4096 normally, 2048 in SAFE MODE.
  • Capped output tokens — num_predict=1024 normally, 256 in SAFE MODE.
  • Model auto-unload — Ollama swaps the inactive model out of VRAM when the active one switches.
  • No third-party dependencies — every pip install is RAM you can no longer use for inference.

The 8GB system prompt

This block is appended to every system prompt sent to either model:

You are running on a low-memory environment (8GB RAM).

Rules:
- prioritize stability over performance
- minimize reasoning depth
- avoid long outputs unless requested
- reduce context usage aggressively
- degrade gracefully under load
- prefer structured short answers
- if overloaded: reduce output length immediately

When SAFE MODE activates, an additional suffix is appended that caps the response at ~120 tokens and disables chain-of-thought.

3 · Self-healing system

Self-healing happens at two layers:

In-process · router.py

  • Watchdog thread (5s tick)
  • SAFE MODE toggle
  • Concurrency semaphore
  • Per-call retry (3 attempts)
  • Fallback chain

Out-of-process · supervisor.py

  • Health-check loop (10s tick)
  • Spawns / respawns router
  • Crash-loop backoff
  • Kills stuck PIDs on :8080
  • Waits for Ollama on boot

Watchdog signals

Every 5 seconds the in-process watchdog refreshes:

SignalSourceUsed for
Host RAM % vm_stat on macOS, /proc/meminfo on Linux SAFE MODE entry / exit
Avg generation latency Rolling window of last 20 requests Latency-spike detection
Recent error count Errors in last 60s SAFE MODE entry
Ollama liveness GET /api/tags (2s timeout) Supervisor restart trigger

Auto-repair rules

  1. Restart router — supervisor detects dead router after 3 × 10s failed health checks.
  2. Kill stuck PIDs — lsof -ti tcp:8080 → SIGTERM then SIGKILL.
  3. Retry Ollama — 3 attempts, 4s apart, before giving up.
  4. Fallback model — if reasoning model errors, retry on coding model with same prompt + system prompt.
  5. Deterministic stub — both models down → return a structured "service degraded" response so callers don't see HTTP errors.
  6. Crash-loop backoff — >5 restarts in 120s → exponential wait (5s × 2ⁿ, capped at 5min).

SAFE MODE behaviour

When SAFE MODE is active, the router automatically:

  • Reduces num_ctx from 4096 → 2048
  • Caps num_predict at 256
  • Drops temperature from 0.7 → 0.5
  • Shortens Ollama timeout from 300s → 120s
  • Appends a SAFE-MODE suffix to the system prompt
  • Rejects requests with 503 if RAM > 95% (critical)

Self-heal log

Every recovery action is appended to logs/self-heal.log in a structured, grep-able format:

4 · Installation & startup

Prerequisites

  • macOS (Apple Silicon) or Linux
  • Python 3.10+
  • Ollama installed and on $PATH
  • (optional) Docker for the security lab targets

1. Pull base models & build the personalities

# pull base models (≈10GB total — done once)
ollama pull qwen3:8b
ollama pull deepseek-r1:8b

# build the HackerStation variants from the Modelfiles
ollama create hackerstation-code   -f Modelfile.qwen-hacker
ollama create hackerstation-reason -f Modelfile.deepseek-hacker

2. Start the stack

# start Ollama + router + supervisor
./start.sh

# individually
./start.sh ollama       # just the model backend
./start.sh router       # just the router (no auto-restart)
./start.sh supervisor   # spawn the self-healing watcher
./start.sh status       # health snapshot
./start.sh stop         # stop supervisor + router cleanly

Why start the supervisor last?

If the supervisor is already running and it sees the router is up, it does nothing. If the router dies, the supervisor spawns it. Always stop the supervisor before the router, or it will helpfully respawn the router you're trying to kill.

3. Verify

curl -s http://localhost:8080/health | python3 -m json.tool
# {
#   "status": "ok",
#   "models": { "coding": "hackerstation-code", "reasoning": "hackerstation-reason" },
#   "safe_mode": false,
#   "memory_percent": 76.2,
#   "ollama_alive": true,
#   "avg_latency_sec": 0.0,
#   "errors_60s": 0
# }

6 · Failure handling examples

Memory overload

What you'll see: latency climbs, then the router starts returning shorter answers, then /health shows "safe_mode": true.

What's happening: the watchdog tripped should_enter_safe_mode() because RAM crossed 88%. num_predict dropped to 256 and the SAFE-MODE suffix is glued onto the system prompt. New requests will be capped until RAM drops below 75% for at least 30s of dwell.

If RAM hits 95%: new requests get a 503 memory_critical response — the router refuses to start work it cannot safely finish.

Ollama crashes

What you'll see: first request to either model returns a fallback response with "_router.fell_back": true; if both models are unreachable, you get the deterministic stub:

{
  "response": "[SERVICE DEGRADED] HackerStation router could not reach any model. ...",
  "model": "deterministic-stub",
  "_router": {"degraded": true, ...}
}

Recovery: the next request will retry Ollama 3× with backoff. Restart Ollama with brew services restart ollama or ./start.sh ollama.

Router stops responding

What you'll see: curl :8080/health hangs or refuses connection.

What happens automatically: the supervisor's router_alive() check fails 3 times in a row, it kills any stuck PID on port 8080 (SIGTERM → SIGKILL), waits for Ollama, and respawns the router. The whole sequence is logged to logs/self-heal.log.

Crash-loop guard: if the router dies more than 5 times in 120 seconds, the supervisor backs off exponentially (5s · 2ⁿ, max 5 min) so it doesn't burn the CPU on a permanently broken build.

7 · Best practices

To keep the box stable on 8GB

  • Use the auto-router (/generate) — it picks the lighter model on ambiguous prompts.
  • Keep the docker lab off while doing heavy AI work (docker compose down).
  • Close other RAM-hungry apps (browsers with many tabs, Slack, IDEs with language servers).
  • Watch ./start.sh status when latency feels off.
  • Tail the self-heal log when iterating on prompts: tail -f logs/self-heal.log.
  • Prefer num_ctx ≤ 4096. Larger context inflates KV-cache RAM linearly.

What NOT to do

  • Don't run two POST /generate calls in parallel — they'll queue on the semaphore anyway, but you'll spike RAM during model swap.
  • Don't disable the supervisor and leave the router running unattended; you lose crash recovery.
  • Don't tighten the SAFE MODE thresholds aggressively (e.g. enter at 70%, exit at 68%) — the router will flap.
  • Don't add heavyweight Python deps (torch, transformers) to router.py. Every MB loaded steals from inference.
  • Don't kill -9 the router while the supervisor is up — stop the supervisor first or it will respawn instantly.

9 · Tuning & config

All knobs are constants at the top of each module. There is intentionally no config file — fewer moving parts, easier to reason about.

ConstantFileDefaultWhat it controls
MAX_BODY_SIZErouter.py 1 MB Max POST body accepted (OOM protection on 8GB)
SEMAPHORE_TIMEOUT_SECrouter.py 90 How long a request waits for the gen lock before 503
OLLAMA_RETRY_ATTEMPTSrouter.py 3 Per-request retries against Ollama
OLLAMA_RETRY_DELAYrouter.py 2.0s Backoff base between retries (multiplied by attempt number)

num_ctx / num_predict (SAFE)router.py 2048 / 256 SAFE MODE Ollama options

MAX_RESTARTS_IN_WINDOWsupervisor.py 5 / 120s Crash-loop protection

The policy block

should_enter_safe_mode() and should_exit_safe_mode() in router.py are the two functions you'll most likely want to tune. They are deliberately separated to enable hysteresis and prevent flapping. Edit, restart the router (the supervisor will respawn it on the same PID slot), and watch logs/self-heal.log to see the effect.
